# シフト管理アプリ（shift-manager）

スタッフのシフトを月間カレンダーで管理するシングルページアプリ。
React + Vite 製で、データは **Googleスプレッドシート**（GAS Web App経由）に保存される。
複数端末・複数スタッフで同じシフト表を共有できる。

## 機能

### 管理者向け
- **スタッフ管理**: 追加・名前変更・削除（色は自動割り当て）、ログイン用メールアドレスと役割（管理者/バイト）の設定
- **シフト種別管理**: 早番・遅番・夜勤・休みをデフォルト搭載。任意の種別（名前・時間帯）を追加可能
  - 日をまたぐ夜勤（例: 22:00〜07:00）も正しく9時間として計算
  - 開始=終了の種別は「休み」などの0時間枠として扱う
- **月間カレンダー**: 日付をクリックして、スタッフ×シフト種別のトグルで割り当て（追加・削除）
- **希望休の承認/却下**: バイトから提出された希望休を一覧確認し、承認・却下できる。承認済みの希望休はカレンダー上にバッジ表示され、シフト割当時にも警告表示
- **月次集計**: スタッフ別の勤務日数・合計時間・種別ごとの回数を自動集計
- **CSV出力**: 表示中の月のシフト表を Excel で開ける CSV（BOM付きUTF-8）でダウンロード
- **シフト希望の取り込み**（任意）: Googleスプレッドシートに集めたシフト希望をGAS経由で読み込み、カレンダー上に参考表示

### バイト向け（閲覧専用）
- 月間カレンダー・月次集計の**閲覧のみ**（シフトの追加・削除はできない）
- **希望休の提出**: 対象日・理由を入力して申請、審査中/承認済み/却下/取消済みの状況を確認できる
- **希望休の取り消し**: 審査中の申請のみ、自分で取り消せる（承認・却下済みは取り消せない）

### 複数店舗運用
- スタッフに「店舗」を割り当てることで、1つのスプレッドシートで複数店舗を管理できる
- 店舗を1件も作らなければ、これまで通り単一店舗として動作する（既存の使い方は変わらない）
- 店舗IDを持たない管理者は「本部管理者」となり、画面上部のセレクトボックスで店舗を切り替えて閲覧・管理できる
- 店舗IDを設定した管理者・バイトは、常に自分の店舗のデータのみを見る
- シフト種別（早番/遅番など）は全店舗共通

### 通知
- 希望休が申請されると、対象店舗の管理者（および本部管理者）へメールで通知
- 希望休が承認/却下されると、申請したスタッフへメールで通知
- GASの `MailApp` を利用（送信数上限は個人のGoogleアカウントで1日100通程度）。送信に失敗しても申請・承認自体は失敗しない
- ブラウザ/スマホのプッシュ通知は、追加のサーバー・VAPID鍵等のインフラが必要なため現時点では未対応

### 共通
- **役割はアカウントで自動判別**: Googleログインしたメールアドレスをスタッフ一覧と照合し、管理者/バイトの画面が自動的に切り替わる
- **スプレッドシートへの自動保存**: 変更は自動的にGAS Web App経由でスプレッドシートに保存される（デバウンス付き）
- **PWA対応**: スマホのホーム画面に追加してアプリのように起動可能

## 重要: このアプリはGASの設定なしには動作しません

スタッフ・シフト種別・確定シフトの保存先はスプレッドシートのみで、ローカル保存のフォールバックはない。
`VITE_GAS_ENDPOINT_URL` / `VITE_GAS_ACCESS_TOKEN` が未設定の場合、アプリは起動画面で
設定を促すメッセージを表示し、それ以上先には進めない。**先に「GASのセットアップ」章を行うこと。**

## 開発

```bash
npm install
npm run dev      # 開発サーバー起動
npm test         # ユニットテスト（vitest）
npm run build    # 本番ビルド（dist/）
```

## データ形式

アプリ内部の状態（`{ staff, shiftTypes, assignments }`）は、GAS Web App経由でスプレッドシートの
以下のシートに読み書きされる（`スタッフ`/`シフト種別`/`シフト表` は初回保存時に自動作成される）：

| シート | 列 | 対応する状態 |
|---|---|---|
| `スタッフ` | id / 名前 / 色 / メールアドレス / 役割 / 店舗ID | `staff: [{ id, name, color, email, role, storeId }]` |
| `シフト種別` | id / 名前 / 略称 / 開始 / 終了 / 色 | `shiftTypes: [{ id, name, short, start, end, color }]` |
| `シフト表` | 日付 / スタッフID / シフト種別ID | `assignments: { "YYYY-MM-DD": [{ staffId, shiftTypeId }] }` |
| `希望休` | id / スタッフID / 対象日 / 理由 / ステータス / 申請日時 / 処理日時 | 希望休の申請・承認/却下/取消ワークフロー（ステータスは pending/approved/rejected/cancelled） |
| `店舗` | id / 店舗名 | `stores: [{ id, name }]` |
| `シフト希望`（読み取り専用・要事前作成） | タイムスタンプ / 氏名 / 対象日 / 希望シフト / 備考 | シフト希望の参考表示のみ |

`スタッフ`シートの「役割」列は `管理者` または `バイト`（未設定時は自動的に「バイト」扱い）。
「メールアドレス」列に、そのスタッフがGoogleログインする際のアカウントのメールアドレスを設定すると、
ログイン時に自動でそのスタッフの役割の画面が表示される。「店舗ID」が空の管理者は「本部管理者」となる。

スタッフ/シフト種別/シフト表の保存は「編集のたびに全件で上書き」方式（複雑な行単位の差分更新はしない）。
GAS側は `LockService` で同時書き込みを直列化するが、厳密なバージョン管理はしていないため、
極端に短い間隔で複数端末から同時編集すると後勝ちになる点に留意する（1店舗規模を想定した設計）。
希望休は申請・承認/却下のたびに1行だけ追加/更新する方式で、他の保存とは独立している。

## GASのセットアップ（必須）

### 1. スプレッドシートを用意する

新規のGoogleスプレッドシートを1つ用意する。`スタッフ`/`シフト種別`/`シフト表` シートは
アプリから初回保存時に自動作成されるため、手動で作る必要はない。

シフト希望を取り込みたい場合は、Google フォームなどで集めたシートを用意し、以下の列を持たせる：

| タイムスタンプ | 氏名 | 対象日 | 希望シフト | 備考 |
|---|---|---|---|---|

- シート名は `シフト希望`（`gas/Code.gs` の `REQUESTS_SHEET` で変更可能）
- 「対象日」は日付型のセルにする
- このシートが無くても他の機能（スタッフ管理・シフト作成）は問題なく使える

### 2. GAS Web App をデプロイする

1. スプレッドシートを開き、拡張機能 > Apps Script
2. `gas/Code.gs` の内容をそのまま貼り付け
3. プロジェクトの設定 > スクリプト プロパティ で `ACCESS_TOKEN` に推測されにくい文字列を設定
4. デプロイ > 新しいデプロイ > 種類「ウェブアプリ」
   - 実行するユーザー: 自分
   - アクセスできるユーザー: 全員
5. 発行された URL を控える

コードを変更した場合は「デプロイを管理」から既存デプロイを更新すること（新規デプロイすると
URLが変わり、アプリ側の環境変数も更新が必要になる）。

### 3. 環境変数を設定する

`.env.example` を `.env` にコピーし、上記で控えた値を設定する：

```bash
cp .env.example .env
```

```
VITE_GAS_ENDPOINT_URL=（手順2のURL）
VITE_GAS_ACCESS_TOKEN=（手順2で設定したACCESS_TOKENと同じ値）
```

設定後は開発サーバーを再起動する（`npm run dev`）。

## Googleログイン・役割分担のセットアップ（推奨）

未設定でもGASさえ設定されていれば通常のシフト管理アプリとして動作する（誰でもアクセス可能・全員管理者相当）。
「管理者」と「バイト」で画面・操作範囲を分けたい場合は、以下を設定する。

1. [Google Cloud Console](https://console.cloud.google.com/) でプロジェクトを作成（または既存のものを使用）
2. 「APIとサービス」>「OAuth同意画面」を設定（社内利用なら「内部」、外部の場合はテストユーザー登録）
3. 「認証情報」>「認証情報を作成」>「OAuthクライアントID」、種類は「ウェブアプリケーション」
4. 「承認済みのJavaScript生成元」に、アプリを公開するURL（例: `https://your-app.vercel.app`、開発時は `http://localhost:5173`）を追加
5. 発行されたクライアントIDを `.env` に追記する：

```
VITE_GOOGLE_CLIENT_ID=（発行されたクライアントID）
```

6. アプリのサイドバー「スタッフ」欄で、各スタッフのメールアドレス（Googleログインに使うもの）と役割（管理者/バイト）を設定する

ログインしたアカウントのメールアドレスがスタッフ一覧のどれとも一致しない場合、そのアカウントは
利用できない（「スタッフとして登録されていません」と表示される）。`VITE_ALLOWED_STAFF_EMAILS` を
併用すると、スタッフ登録の有無とは別に「そもそもログインボタンを押せるメールアドレス」自体を
さらに絞り込むこともできる（任意）：

```
VITE_ALLOWED_STAFF_EMAILS=staff1@example.com,staff2@example.com
```

## セキュリティに関する注意

- このアプリはサーバーを持たない静的サイトのため、`VITE_ALLOWED_STAFF_EMAILS` によるログイン制限はブラウザの開発者ツールで回避され得る「簡易的な目隠し」であり、機密情報の保護には使えない。
- GAS の `ACCESS_TOKEN` も同様にクライアントのコード上に含まれるため、真の秘密情報としては扱えない。第三者に知られた場合は Apps Script のスクリプトプロパティで値を再発行すること。
- `ACCESS_TOKEN` は読み込み時にURLのクエリパラメータとして送信されるため、第三者への漏洩の有無にかかわらず、GAS の「実行数」画面にリクエストURL（トークンを含む）が呼び出しのたびに記録され続ける。より高い機密性が必要な場合は、定期的なトークンのローテーションを検討すること。
- 本格的な認可・監査が必要な規模になった場合は、サーバーサイドでトークンを検証するバックエンドの追加を検討する。
